# 链接、同步、迁移与停用

## 链接 skill

链接 skill 时，要求源目录有效且包含 `SKILL.md`。如果目标名称已经存在：

- 如果它已经是同一个软链接，报告无需修改。
- 如果它是失效软链接，询问是否替换。
- 如果它是真实目录或文件，不要覆盖；建议备份、手动迁移或跳过。

## 批量链接

批量链接多个 skills 时，先逐个验证源目录都包含 `SKILL.md`，再列出计划创建的所有软链接。只在用户确认后执行 `link-many --execute`。

## 删除或停用 skill

删除或停用 skill 时，默认只从当前项目停用，不删除中央原件。先检查目标路径属于用户级还是项目级，以及它是软链接、junction、真实目录、失效链接还是不存在。

- 如果目标是项目级软链接或 junction，只删除链接本身。
- 如果目标是真实目录，停止并说明删除会删除 skill 内容本身，建议迁移、备份或只取消引用。
- 如果用户明确要求删除中央原件，必须列出影响范围，说明所有指向它的项目可能变成失效链接，并要求明确确认口令，例如 `确认删除中央原件 write-blog`。

## 迁移已有真实目录

迁移已有真实目录适用于“全局目录或项目目录里已经复制了一份 skill，现在想改成中央目录 + 软链接”的场景。迁移计划必须说明：

- 真实目录会被移动到中央目录。
- 原位置会变成软链接。
- 不会删除真实 skill 内容。
- 目标中央路径已存在时，默认停止并让用户选择。

如果配置存在且 `default_mode` 是 `centralize`，当当前项目 `.agents/skills` 里出现真实 skill 目录时，不要默认把它当作项目中心原件长期保留；应建议同步或迁移到配置的中央目录，然后在项目中留下软链接。执行前仍必须列出计划并等待用户确认。

## 同步多个 Agent

同步多个 Agent 时，优先使用当前项目 `.agents/skills` 作为项目级入口目录，并且只在用户确认后把 `.claude/skills` 和 `.codex/skills` 链接到它。不要把这个项目级入口目录称为中央 skills 库。

如果其中任一目录已经是真实目录，报告冲突并让用户选择处理方式。不要用软链接覆盖已有的真实目录。

## 共享项目注意事项

对共享或团队项目，要提醒用户：`/Users/name/...` 这类绝对软链接在别人电脑上可能变成失效链接。对团队仓库，尽量使用相对软链接；如果不合适，就提供安装脚本或说明，避免提交绝对软链接。
